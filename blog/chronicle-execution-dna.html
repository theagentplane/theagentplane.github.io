Recording the execution DNA of AI agents

Susheem Koul · June 20, 2026 · 6 min read · Chronicle

Most agent observability tools bolt onto a specific framework: they hook LangGraph nodes, or CrewAI tasks, or a particular vector store client. That works until you change frameworks, mix two of them in the same pipeline, or need to test a fix without re-running a live model. We built Chronicle around a different unit of capture: the decision boundary — any point where your agent calls an LLM, invokes a tool, or makes a routing decision — regardless of what framework sits underneath it.

Why boundaries, not frameworks

A decision boundary is framework-agnostic by definition. Whether your orchestration is LangGraph, a hand-rolled state machine, or a single Python function that calls three tools in sequence, the boundary is the same: something goes in, a model or tool does something non-deterministic or effectful, something comes out. Chronicle wraps that boundary with a single decorator, @boundary, and builds a side execution graph alongside your agent without touching its topology.

Every invocation of a wrapped boundary serializes into an immutable, append-only record we call an Envelope. It captures four things: contextual metadata (pinned model version, sampling parameters, runtime build ID), input state (the full assembled prompt, graph state, retrieved chunks), the action and result (structured tool calls and completions), and graph linkage (parent envelope ID, sequence, invocation index — so retries and branches stay attributable).

The problem this solves

Picture a support agent that issued a $9.8M refund on a $47 order. By the time anyone notices, the conversation that caused it is gone, the model may have been upgraded, and the only evidence is a log line. Without a captured Envelope, you can't tell whether the bug was in the agent's reasoning, the tool's validation, or both — and you can't write a regression test for something you can't reproduce.

With Chronicle, that incident becomes a committed fixture under fixtures/traces/: a full graph of every boundary invocation, replayable on demand. We ship exactly this scenario — along with a currency-mismatch invoice and an oversized trade — as worked examples in the repo.

Two ways to replay

Once an incident is captured, Chronicle's Verification Test Bench gives you two layers of testing:

Layer 1 — deterministic replay

Static envelope and trace fixtures drive structural assertions. This layer never calls an LLM API — it validates control flow and logic against exactly what was recorded, so it's fast, free, and safe to run on every commit.

Layer 2 — LLM-as-judge evaluation

Generation quality isn't a bitwise comparison problem. Layer 2 uses an LLM judge to evaluate meaning — grounding, safety, refusal behavior — against a rubric, for the cases where "did it say the same thing" is the wrong question and "was the output acceptable" is the right one.

Cut-point replay

The most useful pattern in practice is testing a single fix in isolation. You stub everything upstream of the boundary you changed from the saved Envelope, run your new code live at that one boundary, and let everything downstream react to its real output:

session.enable_replay(
    ReplayPlan()
    .stub("agent", 1)          # upstream: frozen from fixture
    .live("delete_file", 1)    # cut-point: run new code
    .live("agent", 2)          # downstream: observe effect
)
result = run_agent(...)
assert session.captured_result("delete_file", 1)["blocked"] is True

That's the whole loop: annotate a boundary once, record an incident when it happens, replay it to understand what went wrong, fix the boundary, and cut-point test the fix against the exact conditions that caused the original failure — without ever calling a live model unless you explicitly ask to.

Chronicle is open source under MIT. The README walks through the financial-incident demos and a deletion-agent example end to end, including an interactive trace visualizer if you want to see the graph rather than read JSON.
