Tokens as infrastructure: why cost controls break in stochastic systems

Susheem Koul · June 16, 2026 · 5 min read · tokenops

Every cloud cost control you've ever configured — budgets, quotas, autoscaling caps, rate limits — was designed for workloads where a request costs roughly the same every time. A web server handles a GET request in a predictable number of CPU cycles. A batch job processes a known number of rows. You can set a budget alert at 80% of expected spend and trust it means something.

Agentic systems don't behave that way, and pretending they do is why teams get blindsided by token spend. That's the starting premise behind tokenops, the field guide we're building for the AI Engineer World's Fair 2026.

Why the old model breaks

A single agent session can vary in cost by an order of magnitude depending on factors that have nothing to do with the input size: how many reasoning steps the model decides to take, whether a tool call fails and triggers a retry loop, whether the agent decides to fan out into sub-agents, or whether a longer context window gets pulled in because a retrieval step returned more chunks than usual. None of that is visible from the request shape alone — it only resolves at execution time, and it can compound across a multi-turn conversation.

Treat that like a deterministic workload and one of two things happens. Set your budget conservatively and you throttle legitimate usage during normal variance. Set it loosely and a single misbehaving session — an agent stuck in a retry loop, or a prompt injection that convinces it to over-explain — can burn through a monthly budget in an afternoon.

The reframe: tokens as infrastructure

tokenops' position is that token consumption deserves the same operational maturity that compute and storage got a decade ago: first-class observability, budgeting primitives built for variance rather than fighting it, and governance that acts at the right layer — per-session, per-agent, per-tool-call — rather than a single blunt monthly cap.

Concretely, that means treating a token budget less like a bank account balance and more like a rate limiter with burst allowance: expect variance, instrument for it, and gate on the outliers rather than the average. It also means cost attribution needs to survive multi-agent fan-out — if agent A spawns agents B and C, the spend has to roll up in a way that's still legible when you're debugging three weeks later.

What's in the guide

tokenops is written as an operational field guide rather than a library — the goal is a reference architecture you can adapt, not a package you install. It covers the failure modes we've hit treating agent workloads like traditional cloud workloads, and what a token-aware infrastructure layer looks like instead.

It's still early and evolving ahead of the AI Engineer World's Fair 2026 (June 29 – July 2, San Francisco) — if you're running agents in production and have hit these problems yourself, the repo is open for issues and PRs.
